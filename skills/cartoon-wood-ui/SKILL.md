# "카툰나무풍" UI 세트 적용 플레이북

> **세트명: 카툰나무풍** (slug `cartoon-wood-ui`, 스타일 앵커 `pb-cartoon-wood-ui`). 사용자가
> "카툰나무풍으로 해줘"라고 하면 이 세트를 쓴다. 다른 UI 세트가 추가되면 같은 방식으로
> `skills/<slug>/SKILL.md` 를 만들고 README 의 "UI 세트 레지스트리"에 등록한다.

Planet Blitz 격납고 파일럿(2026-07-19, PR #62~#65)에서 검증된 **카툰나무풍 세트**를 다른 화면/게임에 적용할 때의 정본. 킷 자산은 이 리포 `library/` 에 캐시 등록돼 있다 — **재생성 금지, `pixellab-cache find` 로 재사용**한다(pixellab 스킬 §4).

## 1) 킷 구성 (library/ 캐시 id · 9-slice 파라미터)

| `pb_ui_icon_*` 16종 | close/coin/crystal/star/lock/salvage/expand/arrow_l·r/gear/upgrade/shield/rocket/check/search/trash | 64² | 통짜 스케일 |

- 렌더는 전부 `scaleMode='nearest'`. 참조 구현: planet-blitz `src/ui/pixi/` (nineSlicePanel·button·slotGrid·tooltip·titleBar·hangar).

## 2) 적용 순서 (승인 게이트 포함)

1. **캐시 조회** — `pixellab-cache find --anchor pb-cartoon-wood-ui` 로 킷 재사용. 새 부품이 필요할 때만 create_ui_asset 으로 **같은 앵커·같은 팔레트** 배치 생성.
2. **목업 승인 게이트** — 구현 전에 실제 자산을 합성한 1920×1080 목업 1장을 만들어 로컬 http 서버로 사용자 브라우저에 열고 승인받는다. 목업에서 확정된 좌표·폰트·간격이 곧 구현 사양이다(아래 §4-9).
3. **구현** — 목업 좌표를 그대로 상수로 옮긴다. 재해석 금지 — 파일럿에서 "구현자가 좌표를 재유도"한 것이 목업 불일치 5건의 근원이었다.
4. **실브라우저 검증** — 반드시 **1920×1080 뷰포트 스크린샷**을 찍어 목업과 나란히 대조한 뒤에 완료를 선언한다. 테스트/빌드 통과 ≠ 시각 검증.

## 3) 함정 1 — 생성 자산에 구워진 텍스트/장식 (최다 재발)

`create_ui_asset` 은 pieces label 을 그대로 **텍스트로 구워 넣는다**("STATS", "CHARACTER INFO" 등). 구워진 텍스트는 9-slice 중앙 스트레치에서 흉하게 늘어난다.

- **버튼/배너/칩**: 반드시 **무지(blank) 자산**을 쓰고 텍스트는 코드로 렌더한다. 신규 생성 시 label 에 `"blank ..., no text"` 를 명시. (`pb_ui_wood_blank_buttons` 가 이렇게 만든 것.)
- **패널**: 중앙에 장식이 구워졌으면 **중앙을 투명 처리**하고, 어두운 내부 채움은 코드로 그린다(프레임보다 먼저 그려 프레임이 덮게). 이때 구멍을 어디서부터 뚫었는지(=살이 끝나는 지점)를 **재지 않고 9-slice 테두리 값으로 갈음하면** §5 의 틈이 생긴다.
- 9-slice 중앙은 "테두리 바로 안쪽 얇은(≤6px) 깨끗한 띠"를 늘리거나 아예 투명+코드 채움으로.

## 4) 함정 2 — 프레임 침범 + **테두리에 붙는 것** (최우선 재발 방지)

나무 프레임(46px)은 시각적으로 두꺼워서 콘텐츠가 1px 만 넘어도 티가 난다. 그리고 **침범하지
않아도 프레임에 딱 붙으면 "테두리에 얹힌 것"으로 보인다** — 사용자가 격납고 파일럿과 연구소
롤아웃에서 **두 번** 지적한 결함이다. `border` 만 피하는 계산은 이 두 번째를 절대 못 막는다.

**규칙: 콘텐츠 상자 = 패널 안쪽으로 `border(46) + pad(14) = 60px`.** 제목·본문·슬롯·버튼·
마스크 하한을 전부 이 상자 기준으로 잡는다. 특히 **패널 제목**은 큰 글자라 프레임에 붙으면
가장 먼저 눈에 띈다 — 제목 top = 상자 top(60)이 기본값이다.

- **좌표를 매 화면 다시 유도하지 말고 헬퍼를 쓴다.** 참조 구현 `planet-blitz src/ui/pixi/nineSlicePanel.ts`:
  `PANEL_BORDER`(46) · `PANEL_INNER_PAD`(14) · `panelContent(w, h) → {x, y, w, h, right, bottom}`.
  화면 코드는 `box.x` / `box.y` / `box.right` / `box.bottom` 만 쓴다. 규칙을 코드로 강제하면
  다음 화면에서 같은 실수가 구조적으로 재발하지 않는다.
- **스크롤 마스크 하한 = `box.bottom`** (패널 높이에서 대충 빼면 셀이 테두리 위에 얹힌다).
- **마스크 높이는 행 단위로 클램프** — `floor(가용높이/행높이)*행높이`. 반토막 셀·반토막 글자가 보이면 사용자는 "삐져나왔다"고 인지한다.
- **슬롯 바깥쪽 라벨**은 라벨 폭(~90px)만큼 컬럼을 안쪽으로: 컬럼 x ≥ `box.x` + 라벨폭 + 12.
- **N행 배치는 간격을 역산**: `step = (box.bottom - 시작y - 셀크기) / (N-1)`. 고정 간격을 먼저 정하면 마지막 행이 걸친다.
- 예외는 **오버레이 한 종류뿐** — 잠금 딤 같은 "안쪽 전체를 덮는 막"은 콘텐츠가 아니므로 여백(14)을 빼고 깐다. 다만 기준은 `border`(46)가 아니라 **살 끝(§5 의 채움 inset)** 이다. 46 으로 깔면 살 끝(37)과 막(46) 사이 8px 이 안 덮여 테두리 안쪽에 밝은 띠로 남는다 — 채움과 똑같은 함정이고, 막은 위에 얹히므로 프레임 밑까지 번져도 안전하다.

## 5) 함정 3 — 코드 채움과 나무 살 사이의 틈 (오버레이에서만 드러난다)

§3 대로 "중앙 투명 + 코드 채움" 을 쓰면 **채움을 얼마나 안쪽에서 시작할지(inset)** 를 정해야
한다. 이 값을 9-slice `border` 에서 유도하면(예: `border - 2` = 44) 채움이 끝나는 자리(44)와
살이 끝나는 자리(37)가 어긋나 **6px 짜리 투명 링**이 남는다. Planet Blitz 에서 실제로 났고,
평면 배경 위 패널에서는 배경색이 비쳐도 티가 안 나 오래 살아남았다 — **화면 위에 얹는
팝업에서만** 뒤 화면이 그대로 비치는 띠로 드러났다(팝업 아래로 뒤 화면 글자가 읽혔고,
설정 팝업 테두리 안쪽으로 뒤 카드의 금색 프레임이 비쳤다).

- **inset 은 자산 알파 실측에서 온다.** 프레임 PNG 를 읽어 ① 가운데 행/열을 가장자리에서
  안쪽으로 훑어 알파가 0 이 되는 지점(=살 끝, 변마다 1px씩 다르니 **제일 얇은 쪽** 기준)
  ② 채움 사각형 경계(모서리 라운드 포함)의 모든 점이 알파 255 위에 있는 최대/최소 inset 을
  구한다. 이 킷 패널은 살 37~39, **안전 구간 6~36**, 채택값 **30**(살 밑으로 7px 겹침).
- **아래로도 한계가 있다.** 프레임 **바깥** 모서리가 둥글어서 inset 을 5 이하로 줄이면 이번엔
  채움 모서리가 프레임 밖으로 삐져나온다. "무조건 0 으로 깔면 안전" 이 아니다.
- **다른 킷·다른 패널이면 다시 잰다.** 살 두께는 그림이 정하지 `border` 가 정하지 않는다.
- **자산 재생성에 대비해 테스트로 묶는다.** 채움은 프레임 밑에 깔려서 값이 어긋나도 코드는
  조용히 돌아간다. PNG 알파를 직접 읽어 상수를 고정하는 단위 테스트를 두면(참조 구현
  `planet-blitz tests/panelFrameGeometry.test.ts`) 시각 검증 전에 깨진다.

## 6) 함정 4 — 레터박스/DPR (Pixi v8)

- `renderer.width/height` 는 **이미 논리(CSS) 픽셀**이다. `resolution` 으로 다시 나누면 Windows 배율(dpr>1)에서 화면 우/하단이 비고 UI 가 축소된다. `app.screen.width/height` 를 쓴다.
- `resizeTo` 경로의 renderer 'resize' 이벤트가 리스너까지 안 오는 환경이 있다 — **window resize 를 직접 구독**하고 rAF 한 프레임 뒤 재적합한다.
- 검증은 dpr>1 환경 + 창 리사이즈 반복으로.

## 7) 함정 5 — 텍스트 선명도

디자인 스페이스(1920×1080)가 축소 표시되면 기본 해상도 Text 는 흐릿/깨져 보인다. **모든 Pixi Text 에 `resolution: 2`** 를 준다. 폰트는 한글 스택("Malgun Gothic", sans-serif), 본문 24/설명 15 이상 + 행 간격 48 이상(빡빡하면 사용자가 바로 지적한다).

## 8) 함정 6 — DOM/캔버스 레이어 혼재

캔버스 UI 화면은 기존 **DOM HUD(HP/LV 등)를 가리지 못한다**(전면 DOM 오버레이 화면들은 자연히 가렸던 것). 캔버스 화면 show/hide 에서 해당 DOM 을 명시적으로 숨기고 복원한다.

## 9) 목업-구현 정합 체크리스트 (완료 선언 전 필수)

- [ ] 프레임 침범 0 (슬롯·라벨·버튼·텍스트 전부)

      찍힌 좌표가 하나도 없는지 코드로 확인하고, 패널 모서리를 확대 크롭해 눈으로도 확인한다
      (사용자 2회 지적 항목 — 스크린샷 전체 보기로는 놓치기 쉽다)
